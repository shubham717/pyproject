SDLC & Agile Interview Prep (Extended)
Date: 2026-03-07

How to use this file
- Read the concepts and definitions first.

- For experience questions, use STAR (Situation, Task, Action, Result).

SDLC Models and When to Use
1) Waterfall
- Sequential phases with fixed scope and heavy documentation.
- Good for stable requirements and regulated environments.

2) V-Model
- Verification and validation mapped to each development phase.
- Good when testing discipline and traceability are required.

3) Iterative/Incremental
- Build in cycles and release in increments.
- Good when early feedback is needed.

4) Spiral
- Risk-driven iterations with prototypes.
- Good for high-risk, complex projects.

5) Agile
- Short iterations, continuous feedback, adaptive scope.
- Good when requirements change frequently.

6) DevOps
- Integration of development and operations with automation and monitoring.
- Good for fast, reliable delivery.

SDLC Core Phases (Common Waterfall View)
1) Requirements
- Gather business/user needs, scope, constraints.

2) Analysis
- Feasibility, risks, and detailed requirement clarification.

3) Design
- Architecture, data models, interfaces, HLD/LLD.

4) Implementation
- Coding, unit tests, code reviews.

5) Testing
- Integration/system/UAT validation.

6) Deployment
- Release to production, rollout, rollback plan.

7) Maintenance
- Support, monitoring, defect fixes, enhancements.

Requirements Concepts
1) Types
- Business, functional, non-functional, user stories, constraints.

2) Requirements quality
- Clear, atomic, testable, traceable, prioritized.

3) Requirements documents
- BRD, PRD, SRS, FRS.

4) Use case vs user story
- Use case: detailed interaction flow.
- User story: brief user value statement.

Testing Levels
1) Unit
2) Integration (API, service, DB)
3) System (end-to-end)
4) Acceptance (UAT)
5) Regression (retest after changes)

Testing Types
1) Functional
2) Non-functional
- Performance (load, stress, soak)
- Security
- Usability and accessibility
- Compatibility (browser/OS/device)
3) Specialized
- Exploratory, Ad hoc
- Smoke and Sanity
- Installation/upgrade
- Localization and i18n

Test Design Techniques
1) Equivalence partitioning
2) Boundary value analysis
3) Decision tables
4) State transition testing
5) Pairwise (orthogonal array)
6) Error guessing

QA and Quality Terminology
1) QA vs QC
- QA: prevention, process focus.
- QC: detection, product focus.

2) Verification vs Validation
- Verification: build the product right.
- Validation: build the right product.

3) Test plan vs test strategy
- Strategy: org or program level direction.
- Plan: project level scope and approach.

4) Test case vs test scenario
- Scenario: high-level objective.
- Case: detailed steps and expected result.

5) Entry/Exit criteria
- Start or stop conditions for testing activities.

Defect Management
1) Defect lifecycle
- New -> Open -> In Progress -> Fixed -> Retest -> Verified -> Closed
- Optional: Reopen, Deferred, Duplicate, Won't Fix, Not Reproducible

2) Severity vs Priority
- Severity: business/technical impact.
- Priority: urgency to fix.

3) Defect leakage vs defect escape
- Leakage: found in later phase.
- Escape: found in production.

4) Root Cause Analysis (RCA)
- Identify why a defect happened and prevent recurrence.

5) Defect triage
- Decide fix order based on risk, impact, effort, release window.

6) Defect metrics
- Defect density, reopen rate, DRE (defect removal efficiency).

Test Documentation and Artifacts
1) Requirements and design docs
- BRD, PRD, SRS, HLD, LLD.

2) Testing docs
- Test strategy, test plan, test cases, RTM, test report.

3) Execution docs
- Test results, defect reports, release notes.

Traceability Matrix (RTM)
- Maps requirements to test cases and ensures coverage.

Risk-Based Testing
- Focus on high impact and high likelihood areas.

Change and Configuration Management
1) Change control
- Formal approval of changes to scope and requirements.

2) Configuration management
- Version control, baselines, build/release tracking.

Environments and Builds
1) Environments
- Dev, QA/Test, Staging/UAT, Production.

2) Build vs Release
- Build: internal version for testing.
- Release: user-facing delivery.

3) Deployment strategies
- Blue/green, canary, rolling, feature flags.

Shift-Left and Shift-Right
1) Shift-left
- Test earlier (requirements reviews, unit and API tests).

2) Shift-right
- Test in production (monitoring, A/B tests, logging).

Agile Concepts (Expanded)
Agile Values
- Individuals and interactions over processes and tools.
- Working software over comprehensive documentation.
- Customer collaboration over contract negotiation.
- Responding to change over following a plan.

Agile Principles (summary)
- Deliver value early and often, welcome change, continuous improvement.

Scrum Roles
1) Product Owner
- Owns backlog, prioritizes value, clarifies requirements.

2) Scrum Master
- Facilitates Scrum, removes impediments, coaches team.

3) Developers
- Build, test, and deliver the increment.

Scrum Events
1) Sprint Planning
- Define sprint goal and select backlog items.

2) Daily Scrum
- 15-minute sync on progress and impediments.

3) Sprint Review
- Demo increment, gather feedback.

4) Sprint Retrospective
- Inspect and improve the process.

Scrum Artifacts
1) Product Backlog
2) Sprint Backlog
3) Increment

Agile Planning and Estimation
1) Estimation methods
- Story points, T-shirt sizes, ideal hours.
- Planning poker, affinity estimation.

2) Velocity
- Average completed story points per sprint.

3) Capacity vs Velocity
- Capacity: available effort for a sprint.
- Velocity: historical delivered points.

User Stories and Acceptance Criteria
1) User story template
- As a [user], I want [goal], so that [benefit].

2) INVEST criteria
- Independent, Negotiable, Valuable, Estimable, Small, Testable.

3) Acceptance criteria formats
- Given/When/Then or checklist style.

Definition of Ready (DoR)
- Clear, small, testable, acceptance criteria defined, dependencies known.

Definition of Done (DoD)
- Code complete, tests passing, review done, documentation updated.

Agile Metrics
1) Burndown and burnup
2) Lead time vs cycle time
3) WIP limits (Kanban)
4) Throughput

Agile Frameworks (beyond Scrum)
1) Kanban
- Continuous flow, visual board, WIP limits.

2) Extreme Programming (XP)
- Pair programming, TDD, continuous integration.

3) Lean
- Eliminate waste, optimize flow.

4) SAFe (Scaled Agile)
- Large enterprise scaling framework.

Agile Testing Concepts
1) Agile testing quadrants
- Q1: unit/component, developer tests.
- Q2: functional/story tests.
- Q3: exploratory/usability tests.
- Q4: performance/security tests.

2) CI/CD
- Continuous Integration, Continuous Delivery/Deployment.

3) Automation pyramid
- Unit > API/service > UI tests.

4) Test automation strategy
- What to automate, where, and how to maintain.

Release and Roadmapping
1) Release planning
- Scope, timeline, capacity, risks.

2) MVP (Minimum Viable Product)
- Smallest release to learn from users.

3) Technical debt
- Short-term tradeoffs that slow future delivery.

4) Spikes
- Time-boxed research to reduce uncertainty.

Agile vs Waterfall (Quick Compare)
- Agile: iterative, adaptive, frequent feedback.
- Waterfall: sequential, fixed scope phases.

Quick Reference: One-line Definitions
- Acceptance criteria: conditions a story must meet to be accepted.
- Backlog: ordered list of work for the product.
- Burndown chart: remaining work across the sprint timeline.
- CI/CD: automated build, test, and release pipeline.
- DoD: shared completion checklist.
- DoR: shared readiness checklist.
- Increment: shippable product at sprint end.
- MVP: smallest product to learn from users.
- RTM: mapping from requirements to tests.
- Spike: time-boxed research task.
- Sprint: fixed-length iteration.
- Story points: relative effort estimate.
- Technical debt: tradeoffs that increase future cost.
- Velocity: completed points per sprint.
